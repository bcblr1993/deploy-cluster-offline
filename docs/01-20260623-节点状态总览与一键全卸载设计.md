## 1. 目标与定位

6 步部署向导走完后，运维需要一个**总览主页**，随时回答三个问题：

1. 每台节点现在**通不通、状态如何**（连通性、主机名、docker 是否在跑、负载/磁盘概览）。
2. 每台节点上**正在运行哪些服务**（容器状态、端口、是否本工具部署的、集群健康）。
3. 想推倒重来时，能**一键卸载所有节点的所有服务**（危险操作）。

定位：这是部署完成后的**运维态主页**（Operations Overview），与"部署向导"是两种模式。向导是"从零装"，总览是"看与清"。

---

## 2. 入口与导航

- **向导第 6 步部署成功后**，自动给出"进入运维总览"的入口。
- 顶栏常驻一个 **「运维总览」** 入口，任何时候可切入（不依赖是否刚部署）。
- 总览页可"返回向导"继续调整。
- 进入总览不要求走完向导：只要步骤1 有可连节点即可查看（容器发现基于真实 `docker ps`，与向导放置矩阵解耦）。

> 实现上：在现有 `App.tsx` 的"向导"之外增加一个 `view` 维度（`wizard` | `overview`），顶栏切换。不破坏现有向导结构。

---

## 3. 信息架构 / 布局

页面 = 顶部汇总条 + 节点卡片网格。

```
┌── 汇总条：节点 N 台(在线 X) · 运行容器 Y · [刷新] [自动刷新] [一键全卸载(danger)] ──┐

┌─ 节点卡片: node-1 (10.8.8.71) ───────────┐  ┌─ 节点卡片: node-2 ──────────┐
│ ● 在线  root  Ubuntu 22.04  x86_64        │  │ ● 在线 ...                   │
│ docker active · load 0.3 · / 使用 42%     │  │                              │
│ ┌ 服务/容器 ────────────────────────────┐ │  │ ...                          │
│ │ ● postgres   Up 2h   5432   [日志][停] │ │  │                              │
│ │ ● kafka      Up 2h   9092   集群:OK    │ │  │                              │
│ │ ● cassandra  Up 2h   9042   UN         │ │  │                              │
│ │ ● iotcloud   Up 1h   8080              │ │  │                              │
│ └────────────────────────────────────────┘ │  │                              │
└──────────────────────────────────────────┘  └──────────────────────────────┘
```

- **节点卡片**：主机名/IP/用户、连通状态、OS/架构、docker 是否 active、负载与根分区使用率（轻量概览）。
- **服务/容器列表**：每行一个容器——状态点（运行/退出/重启中）、容器名、对应服务（按名映射到 catalog）、`Up 时长`、对外端口。本期**无行内操作**（只看，不单独停/启）。
- **外部容器**（非本工具部署）：标灰显示"外部"，**只读、无任何操作权限**，不纳入一键卸载。

## 4. 数据来源与状态探测

每台节点并发探测（复用 `SshPool`）：

| 数据 | 命令（示意） |
|---|---|
| 连通/主机名/OS/架构 | 复用步骤1 探测（`hostname`、`/etc/os-release`、`uname -m`） |
| docker 是否在跑 | `systemctl is-active docker` |
| 负载 / 根分区 | `uptime`、`df -P / \| tail -1` |
| 运行容器 | `docker ps --format '{{.Names}}\t{{.Status}}\t{{.Ports}}'`（sudo） |
| 集群健康(可选) | cassandra `nodetool status`、kafka 端口可达 |

**服务识别**：把容器名（`postgres`/`redis`/`kafka`/`cassandra`/`iotcloud`/`netdata`/`wechat-messenger-container`）映射回 `services/catalog.ts`，未知名归为"外部容器"。

> 容器发现以**真实 `docker ps` 为准**，不依赖工程里上次的放置矩阵——这样即便是上一次会话或手工部署的服务也能看到、能清理。

---

## 5. 刷新机制

- **手动刷新**按钮（默认）。
- **自动刷新**开关：开启后每 ~10s 轮询一次（可配）；切走总览或执行卸载时暂停，避免无谓 SSH 压力。
- 单节点探测失败只标红该卡片，不影响其它节点（失败隔离）。

---

## 6. 一键全卸载（危险操作）

**范围**：所有**可连节点**上、**本工具管理的**服务容器。

**流程**：
1. 先探测各节点本工具管理的容器（按 §4 识别 + 解析其部署目录 `~/sprixin-iotcloud/services/<svc>`）。
2. 生成 `ActionPlan`（§15，**danger** 级）：列出每节点将停止/删除的容器；可选"同时删除数据卷"；要求输入确认词（如 `UNINSTALL-ALL`）。
3. 执行：**倒序**（应用层 iotcloud/wechat → 中间件 kafka/cassandra/redis → 存储 pg），逐目录 `docker-compose down [-v]`；目录不存在则跳过。
4. 复用 §20 的卸载逻辑与事件流（`RunEvent`），总览页内嵌实时日志/状态。

**安全**：
- 默认**保留数据卷**；删卷需显式勾选 + 确认词。
- 只动识别为本工具服务的容器，**不碰外部容器**。
- 执行期间沿用全局 `busy` 锁，禁用刷新与导航切换。

> 与现有"步骤6 内的卸载"区别：那个按"当前放置矩阵"卸载；本功能按"真实发现的容器"全量卸载，覆盖面更广，更适合"推倒重来"。

## 8. 数据模型（草案）

```ts
interface ContainerInfo {
  name: string
  service?: ServiceId   // 映射到 catalog；undefined=外部容器
  status: string        // docker ps 的 Status 原文，如 "Up 2 hours"
  state: 'running' | 'exited' | 'restarting' | 'unknown'
  ports?: string
}

interface NodeStatus {
  reachable: boolean
  hostname?: string
  osPretty?: string
  arch?: string
  dockerActive: boolean
  load1?: number
  rootUsedPercent?: number
  containers: ContainerInfo[]
  error?: string
}
```

## 9. 边界与风险

- **部分节点不可连**：总览正常展示其余节点；一键全卸载只作用于可连节点，并提示哪些被跳过。
- **外部容器**：识别但默认不清理，避免误删现场其它业务。
- **自动刷新与执行冲突**：执行卸载/操作时暂停轮询。
- **大集群**：容器探测命令轻量；自动刷新间隔可调，避免 SSH 拥塞。
- **删数据卷不可逆**：danger 级 + 确认词 + 预览受影响项。
