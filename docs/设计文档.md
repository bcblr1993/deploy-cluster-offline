# 离线集群部署工具 — 设计文档

> 版本：v0.1（待审阅）
> 最近更新：2026-06-22
> 关联产物：`sprixin-iotcloud-4.1.0-20260530.1-installer.tar.gz`（现场 iotcloud 安装包）

---

## 0. 文档目的

本工具用于在**离线现场**把一套 iotcloud 平台部署到**多台物理机（节点）** 上。运维人员在自己的笔记本上打开本工具，通过 SSH 远程操控所有现场机器，按向导一步步完成：主机接入 → 主机名/hosts → 时间对齐 → 磁盘预览 → 离线安装 Docker → （后续）按节点选服务部署。

本文档为**实施前的设计基线**，确认后再进入编码。

---

## 1. 需求与范围

### 1.1 功能需求（按向导步骤）

| 步骤 | 名称 | 核心动作 | 本期范围 |
|---|---|---|---|
| 1 | 主机配置 + 连接检测 | 录入多台主机(IP/端口/用户名/密码)，并发检测连通性与环境 | ✅ 本期 |
| 2 | 主机名 & hosts 映射 | 设置各机 hostname，全集群互相写 hosts | ✅ 本期 |
| 3 | 时间对齐 | 按网络连通情况自动选源，保证全集群时间一致 | ✅ 本期 |
| 4 | 磁盘分区预览 | 列出每台机磁盘类型(SSD/HDD)、分区、容量、挂载、使用率 | ✅ 本期 |
| 5 | 离线安装 Docker / docker-compose | 从安装包抽取二进制下发并安装，幂等校验 | ✅ 本期 |
| 6 | 按节点选服务部署 | 选节点 → 分发镜像/compose → 起服务（含跨节点集群编排） | ✅ 已细化（见 §16） |

### 1.2 非功能需求

- **跨平台**：运维端支持 Windows / macOS / Linux 三端。
- **离线**：不依赖外网，安装物料全部来自本地安装包。
- **无 Agent**：现场机器只需开 SSH，不预装任何东西。
- **幂等**：每一步可重复执行不报错、不污染系统配置。
- **多架构**：现场可能混有 x86_64 与 ARM(aarch64) 机器，需按架构分发。
- **并发**：所有步骤对 N 台机并发执行，单机独立显示状态/日志。
- **凭据安全**：主机密码本地加密保存，明文绝不落盘。

### 1.3 明确不在本期范围

- 步骤 6 服务编排（cassandra/kafka/redis-cluster 跨节点组网）——前 5 步定稿后单独出设计。
- 集群运维态（监控、升级、扩缩容、回滚）——属后续迭代。
- 密钥(免密)登录——预留接口，本期先做密码登录。

## 2. 安装包分析（约束来源）

解包 `sprixin-iotcloud/` 结构与对工具的意义：

```
sprixin-iotcloud/
├─ docker/                       # 离线 docker 物料（本工具步骤5直接复用）
│  ├─ bin/                       # dockerd / docker / docker-compose / containerd / runc / ctr ...
│  ├─ service/docker.service     # systemd 单元
│  └─ daemon.json                # { "data-root": "/root/.docker" }
├─ images/                       # 7 个服务镜像 tar（步骤6 docker load 用）
│  ├─ redis.tar postgres.tar cassandra.tar kafka.tar
│  ├─ iotcloud-20260530.1.tar netdata.tar wechat-messenger-v2.1.0.tar
├─ services/                     # 每服务一份 compose（步骤6 用）
│  ├─ cassandra/ kafka/ postgres/ redis/ redis-cluster/
│  ├─ netdata/ wechat-messenger-v2.1.0/ iotcloud/
├─ install.sh                    # 现有单机安装脚本（本工具的分布式化蓝本）
├─ start.sh                      # 现有单服务 up/down/restart 脚本
└─ uninstall.sh
```

**关键约束：**

1. **架构**：`docker/bin/dockerd` 等为 **x86-64 / Linux / 静态编译**。ARM 机器不可用此二进制，需要另一份 `arm64` 安装包（见 §6 多架构设计）。
2. **systemd**：`docker.service` 走 systemd，目标系统需 systemd 体系（CentOS7+/Ubuntu16+/麒麟/统信等满足）。
3. **data-root**：`daemon.json` 默认数据目录 `/root/.docker`，与磁盘步骤联动可改到大容量 SSD。
4. **install.sh 逻辑可复用**：离线装 docker → 装 docker-compose → 建 docker 组 → 建 `proxy` 网络 → load 镜像 → 起 iotcloud。本工具把它**拆分到对应向导步骤**并分布式/幂等化。

## 3. 技术选型

| 维度 | 选型 | 理由 |
|---|---|---|
| 桌面框架 | **Electron** | 生态成熟、三端打包稳定、SSH 库选择多 |
| 前端 | **React + TypeScript + Vite** | 组件化、类型安全、热更新快 |
| UI 组件库 | **Ant Design** | 步骤条/表单/表格/进度/日志组件齐全，契合运维向导 |
| 前端状态 | **Zustand** | 轻量，适合向导式分步状态 |
| SSH/SFTP | **ssh2（Node 原生实现）** | 纯 JS，不依赖本机/目标机 ssh 客户端，Windows 友好 |
| 凭据存储 | **Electron `safeStorage`** | 内置，底层走 macOS Keychain / Win DPAPI / Linux libsecret |
| 打包 | **electron-builder** | 出 Win(NSIS .exe) / macOS(.dmg) / Linux(AppImage/.deb) |

**为何不选 Tauri**：包体确实更小，但 SSH 要用 Rust russh，开发成本与踩坑更高；本工具核心价值在“可靠的远程编排”，Node 的 ssh2 生态更利于快速做稳。

## 4. 系统架构

### 4.1 进程模型

```
┌─────────────────────────── Electron App ───────────────────────────┐
│  渲染进程 (Renderer / React)            主进程 (Main / Node.js)       │
│  ┌───────────────────────┐   IPC      ┌──────────────────────────┐  │
│  │ 向导 UI                │ <───────>  │ SSH 连接池 SshPool        │  │
│  │  Step1..Step6 页面     │  (invoke   │ SFTP 传输器 FileTransfer  │  │
│  │  节点状态表 / 日志流    │   + event) │ 任务编排引擎 Orchestrator │  │
│  │  Zustand store         │            │ 安装包管理 PackageManager │  │
│  └───────────────────────┘            │ 配置/凭据 ConfigStore     │  │
│                                        └────────────┬─────────────┘  │
└─────────────────────────────────────────────────────┼───────────────┘
                                                       │ ssh2 (exec + sftp)
                          ┌────────────────────────────┼────────────────┐
                          ▼                ▼            ▼                ▼
                      节点1(x86_64)    节点2(x86_64)  节点3(aarch64) ... 节点N
                       仅需开放 SSH，无需预装任何 agent
```

- **渲染进程**：纯 UI + 本地状态，**不直接碰网络/SSH**，所有远程操作通过 IPC 委托主进程。
- **主进程**：承载所有有状态/有副作用的能力（SSH、文件、磁盘读写、加密）。
- **通信**：命令用 `ipcRenderer.invoke`（请求-响应）；长任务的实时日志/进度用主进程 `webContents.send` 事件流推回。

### 4.2 核心模块职责

| 模块 | 职责 | 关键点 |
|---|---|---|
| `SshPool` | 维护每个节点的 SSH 连接，提供 `exec(cmd)` / `execSudo(cmd)` | 连接复用、超时、keepalive、并发上限 |
| `FileTransfer` | SFTP 上传，带进度回调、断点续传 | 大文件(后续镜像 GB 级)分片、校验和 |
| `Orchestrator` | 把“一步操作”抽象为对 N 节点的并发任务，统一状态机 | 幂等、重试、失败隔离、事件流 |
| `PackageManager` | 解析/缓存安装包，按架构定位物料 | 多架构、抽取 `docker/` 子集免传全量 |
| `ConfigStore` | 持久化集群配置；凭据用 safeStorage 加密 | 敏感/非敏感分离 |
| `steps/*` | 每个向导步骤的远程命令编排逻辑 | 见 §7 |

### 4.3 任务编排引擎（核心抽象）

每个步骤都是一个 **Step**，对所有节点并发跑同一套逻辑。统一状态机：

```
pending → running → (success | failed)
                          └─ retry → running
```

引擎对外提供：
- `runStep(stepId, nodes, params)`：并发执行，返回每节点结果。
- 事件流：`node:log`（实时输出行）、`node:status`（状态切换）、`node:progress`（百分比，用于传输）。
- **失败隔离**：某节点失败不阻断其它节点；UI 单独标红并支持“仅重试失败节点”。
- **幂等保证**：每个 Step 的远程命令都设计成可重复执行（详见各步骤）。

## 6. 多架构设计（现场可能混有 ARM）

当前安装包只有 x86_64 物料，ARM 机器无法使用。设计原则：**架构差异在远端，按节点选包。**

1. **安装包来源可配多份**：UI 支持登记「x86_64 安装包」「arm64 安装包」各自路径。`PackageManager` 解析每份的架构（读 `docker/bin/dockerd` 的 ELF 头或包内 manifest）。
2. **步骤1 探测架构**：`uname -m` → `x86_64` / `aarch64`。
3. **按节点匹配物料**：步骤5/6 为每台机自动选对应架构的 `docker/` 与 `images/`。
4. **缺包早提示**：若现场有 aarch64 节点但未登记 arm64 安装包，**步骤1 即标红**“需要 arm64 安装包”，不拖到安装阶段才失败。
5. **镜像同理**：步骤6 的 `docker load` 也要架构匹配（arm64 镜像需单独构建/导出）。

> **本期结论（§14-1）**：暂无 arm64 包，ARM 线只「预留接口」——多架构抽象与检测保留，aarch64 节点在步骤1 拦截提示“缺 arm64 安装包”，x86_64 节点完整跑通。

## 7. 各步骤详细设计

> 约定：所有需要 root 的命令统一经 `execSudo`（root 直连则裸跑；普通用户 `sudo -S` 读密码）。每条命令均按幂等设计。

### 步骤 1：主机配置 + 连接检测

**UI**：可增删的主机表格（IP / 端口 / 用户名 / 密码 / 备注），“检测全部”按钮。

**远程探测命令（并发，每台一连接）**：
```bash
uname -m                                   # 架构
. /etc/os-release 2>/dev/null; echo "$PRETTY_NAME"   # 发行版
[ -d /run/systemd/system ] && echo systemd # systemd 判定
id -u                                       # 0=root
sudo -n true 2>/dev/null && echo sudo-ok    # 免密 sudo? 否则需密码
command -v docker >/dev/null && echo docker-installed
```
**判定规则**：
- 连不通 → 红，显示原因（认证失败/超时/拒绝）。
- 架构非 x86_64/aarch64，或非 systemd → 红，给出说明。
- 架构是 aarch64 但缺 arm64 安装包 → 黄/红，提示补包。
- 权限既非 root 又不能 sudo → 红，无法继续。

**门禁**：所有节点通过校验才允许“下一步”。

**边界情况**：sudo 需要 tty（`requiretty`）→ 用 `ssh2` 申请 pty + `sudo -S` 喂密码；密码错误明确提示。

### 步骤 2：主机名 & hosts 映射

**UI**：为每台机分配 hostname（默认建议 `node-1..node-N`，可改，校验合法性/唯一性）。

**远程命令**：
```bash
# 1) 设置主机名（幂等）
hostnamectl set-hostname <hostname>

# 2) 写全集群 hosts，用标记块保证幂等（先删旧块再追加新块）
sed -i '/# === deploy-tool BEGIN ===/,/# === deploy-tool END ===/d' /etc/hosts
cat >> /etc/hosts <<'EOF'
# === deploy-tool BEGIN ===
10.0.0.11 node-1
10.0.0.12 node-2
...（全集群每个 IP→hostname）
# === deploy-tool END ===
EOF
```
**要点**：每台机都写入**全集群**映射（含自己）；标记块隔离，反复执行只替换块内内容，不动用户原有条目。

### 步骤 3：时间对齐

**目标**：保证全集群时间一致（首要），尽量贴近真实时间（次要）。统一用 **chrony**（离线收敛稳、好控）。

**判定流程**：
```
1. 并发检测每台外网连通性：
   timeout 3 bash -c "echo > /dev/tcp/ntp.aliyun.com/123" 或 ping 公网
2. 分支：
   A. 全部能上网  → 各机 chrony 指向公网 NTP 池，独立同步
   B. 部分能上网  → 选一台“能上网”的做时间源(server，并 allow 内网段；自身对上游公网)，
                    其余节点 chrony server 指向该源
   C. 都不能上网  → 选一台做“基准源”(local stratum 10)，其余指向它；
                    保证内部一致（即使整体偏离真实时间）
3. 同步后回读每台 `date +%s`，计算两两偏差，UI 展示收敛结果（阈值如 <1s 视为已对齐）
```
**远程命令（示意）**：
```bash
# chrony 由工具内置离线包下发安装（§14-2，不依赖系统自带）：
#   检测未装 → SFTP 下发对应发行版的 chrony 离线包(rpm/deb) → 本地安装
# 写 /etc/chrony.conf：server/pool + (源机) allow <subnet> + local stratum 10
systemctl enable --now chronyd
chronyc makestep        # 立即步进对齐
chronyc tracking        # 回读状态
```
**边界情况**：
- chrony 离线包需按发行版/包格式准备（rpm/deb，见 §18 物料清单）；本期仅 x86_64。
- 极端兜底：若目标发行版无匹配 chrony 包，退化为 `date -s` 拉齐 + `hwclock -w`（一致性可保，精度略低）。
- 时区不一致 → 统一 `timedatectl set-timezone Asia/Shanghai`（与 install.sh 一致）。

### 步骤 4：磁盘分区预览（只读）

**远程命令**：
```bash
lsblk -dbno NAME,SIZE,ROTA,TYPE,MODEL        # 整盘：ROTA=0→SSD, 1→HDD
lsblk -bno NAME,FSTYPE,SIZE,MOUNTPOINT,TYPE  # 分区树
df -B1 --output=target,pcent 2>/dev/null     # 挂载点使用率
```
**展示**：按节点分组的表格——盘符 / 类型(SSD/HDD) / 容量 / 型号 / 分区(文件系统/容量/挂载点/使用率)。本步**只读不改**。

> 注：Docker `data-root` 本期固定默认 `/root/.docker`（§14-4），此步不做数据盘选择；磁盘预览主要供运维判断容量、为步骤6 各服务**数据卷落盘路径**（§16.4）提供参考。

### 步骤 5：离线安装 Docker / docker-compose

**物料准备（本机）**：`PackageManager` 从安装包**只抽取 `docker/` 子目录**（几百 MB，不传 1.5G 镜像），按节点架构选对应包。

**传输**：`FileTransfer` SFTP 并发上传到各机 `/opt/deploy-tool/docker/`，带进度条。

**远程安装（复刻 install.sh 离线分支 + 幂等）**：
```bash
# 已装则跳过
if command -v docker >/dev/null; then echo skip; else
  cp /opt/deploy-tool/docker/bin/* /usr/bin/
  cp /opt/deploy-tool/docker/service/docker.service /etc/systemd/system/
  mkdir -p /etc/docker
  cp /opt/deploy-tool/docker/daemon.json /etc/docker/   # data-root 固定默认 /root/.docker（§14-4）
  chmod +x /usr/bin/docker*
  systemctl enable docker; systemctl daemon-reload; systemctl restart docker
fi
# docker-compose（包内自带二进制）
cp /opt/deploy-tool/docker/bin/docker-compose /usr/bin/ && chmod +x /usr/bin/docker-compose
# docker 组 + proxy 网络（幂等）
getent group docker || groupadd docker
docker network inspect proxy >/dev/null 2>&1 || docker network create proxy
```
**校验**：回读 `docker version` 与 `docker-compose version`，任一失败标红并展示日志。

**已装环境处理（§14-13，目标机可能装过 docker/旧服务）**：步骤1/5 检测到已装 docker 时，提供两种模式：
- **复用（默认，info 级）**：跳过安装，直接用现有 docker；仅补齐缺失项（compose、proxy 网络、docker 组）。
- **强制重装（danger 级，§15）**：停止并卸载现有 docker、**删除其下容器/镜像/卷与 `data-root` 数据**，再离线重装。确认弹窗须列出每台将被清除的内容并要求**输入确认词**。
此外步骤6 前会探测各节点是否已存在同名容器（如旧 `cassandra`/`kafka`），提示“将覆盖/重建”（danger，见 §20）。

**边界情况**：已装 docker 但版本不符（强制重装可统一版本）/ `data-root` 磁盘空间不足 / SELinux 影响（install.sh 里有注释处理，按需开关）。

### 步骤 6：按节点选服务部署

详见 **§16 步骤6 服务编排详细设计**。

## 8. 凭据与配置持久化

- **敏感数据（密码）**：经 `safeStorage.encryptString` 加密后存盘，按 `nodeId` 索引；读取时 `decryptString`。底层 macOS Keychain / Win DPAPI / Linux libsecret。
- **非敏感数据**：集群拓扑、主机名、安装包路径、步骤进度等存普通 JSON。
- **文件位置**：`app.getPath('userData')/projects/<project>.json`（非敏感）+ 加密凭据区。
- **多工程**：支持保存多个现场为不同工程，下次打开免重录。

## 9. 错误处理 / 日志 / 可观测性

- **统一日志流**：每节点每步的 stdout/stderr 实时推到 UI 日志面板，可按节点筛选、可导出。
- **结构化结果**：每步返回 `NodeTaskResult[]`，UI 用颜色/图标表达状态。
- **重试**：失败节点支持“仅重试失败项”，不影响已成功节点。
- **超时与中断**：SSH/命令级超时；用户可中断长任务（传输/安装）。
- **本地审计日志**：操作落地到本机日志文件，便于现场复盘。

## 15. 危险操作确认机制（贯穿全流程）

**原则：任何会变更/覆盖/删除远端状态的操作，执行前必须弹出确认，清楚列出“将在哪些节点做什么、会覆盖/删除什么”，用户确认后才执行。** 只读探测（步骤1 检测、步骤4 磁盘预览）不弹确认。

### 15.1 风险分级

| 级别 | 含义 | 交互 |
|---|---|---|
| `info` | 新增/幂等写入，基本无损 | 普通确认弹窗，列出变更项 |
| `warning` | 覆盖既有配置（可能影响现状） | 确认弹窗 + 红色高亮受影响项 |
| `danger` | 删除数据/卸载组件/重置系统态 | 确认弹窗 + **要求手动勾选/输入确认词**（如输入 `DELETE`）才放行 |

### 15.2 各步骤的确认点

| 步骤 | 操作 | 级别 | 确认弹窗要点 |
|---|---|---|---|
| 2 | 改 hostname、写 `/etc/hosts` | warning | 列出每台“旧→新主机名”、将写入的 hosts 块内容 |
| 3 | 改时间/时区、改 chrony 配置 | warning | 列出时间源策略、哪台做源、各机时间将被步进调整 |
| 5 | 安装 docker | info/warning | 列出将写入 `/usr/bin`、`/etc/docker`、systemd 单元的文件 |
| 5 | **强制重装（清除已有 docker）** | **danger** | **明确列出将停止并删除现有 docker、容器、`data-root` 数据的节点；要求输入确认词** |
| 6 | 起服务、`docker load`、写 compose | warning | 列出每台将新增的容器、占用端口、数据卷路径 |
| 6 | **重新部署/卸载服务（down + 删卷）** | **danger** | **列出将销毁的容器与数据卷；要求输入确认词** |

### 15.3 实现方式

- 统一 `confirmAction(plan: ActionPlan)`：每个 Step 在真正执行前先生成 `ActionPlan`（结构化的“将做什么”清单：节点 → 动作 → 受影响文件/数据/端口 + 风险级别），由前端统一渲染确认弹窗。
- `danger` 级别弹窗禁用“回车直接确认”，必须显式勾选“我已了解风险”或输入确认词。
- 确认内容与最终执行命令**同源**（由同一份 plan 生成），避免“提示与实际不符”。
- 所有确认与执行结果写入本机审计日志。

```ts
interface ActionPlan {
  stepId: string;
  level: 'info' | 'warning' | 'danger';
  items: {
    nodeId: string;
    summary: string;            // “node-01: 覆盖 /etc/hosts 托管块”
    affects: string[];          // 受影响文件/数据/端口
    destructive?: string[];     // 将删除的内容（danger 用）
  }[];
  requireKeyword?: string;      // danger 时需用户输入的确认词
}
```

## 16. 步骤 6 服务编排详细设计

目标：用户像搭积木一样，把任意服务实例放到任意节点上（示例：node-01 放 pg/redis/iotcloud/wechat/kafka-01/cassandra-01，node-02 放 kafka-02/cassandra-02，node-03 放 kafka-03/cassandra-03），工具自动完成**镜像按需分发 + 跨节点地址装配 + 集群参数计算 + 按依赖顺序启动 + 健康校验**。

### 16.1 核心难点与解法：跨节点网络

现有 compose 假设“单机同 `proxy` 网络、用容器名互连”（如 `CASSANDRA_URL: cassandra:9042`），**容器名 DNS 跨不了机器**。解法：

1. **对外暴露真实端口**：每个服务通过宿主机端口发布（compose 已 `ports: 9042:9042` 等），其它节点用 `节点IP:端口` 即可访问。
2. **依赖地址装配**：消费方（iotcloud）的依赖地址由工具改写为依赖实例**所在节点的真实 IP:端口**，不再用容器名。
3. **集群内部通信用 `host` 网络**：cassandra(gossip 7000)、kafka(controller 9093 + advertised)、redis-cluster(bus 端口) 这类需要“对端用真实地址回连”的，采用 `network_mode: host` + 公告真实 IP，避免 NAT/桥接导致的回连失败。
4. **沿用步骤2 的 hosts 映射**：装配时可用 `node-01` 主机名（已写入各机 `/etc/hosts`），host 网络容器能直接解析；桥接容器则注入 IP。

### 16.2 服务目录（Service Catalog）

工具内置一份服务元数据（驱动放置规则、参数生成、镜像匹配）：

| 服务 | 镜像 | 角色 | 实例模型 | 关键端口 | 数据卷(可选路径) | 依赖 | 跨节点要点 |
|---|---|---|---|---|---|---|---|
| postgres | postgres:15.4 | 关系库 | 单例 | 5432 | /var/lib/postgresql/data | — | 被 iotcloud 连接 |
| redis | redis:7.2 | 缓存 | 单例 | 6379 | /data | — | 被 iotcloud 连接 |
| redis-cluster | redis:7.2 | 缓存集群 | 多实例(≥3 主) | 700x + bus | /data/redisN | — | host 网络、announce IP、需 init |
| kafka | bitnami/kafka:3.9 | 消息 | 单例 / KRaft 集群 | 9092,9093 | /bitnami/kafka | — | node-id、quorum voters、advertised |
| cassandra | cassandra:4.1.3 | 时序库 | 单例 / 集群 | 9042,7000 | /var/lib/cassandra | — | seeds、listen address |
| iotcloud | iotcloud:4.1.0 | 应用 | 单例(本期) | 1883,8080,7070,7071,5683-5688/udp | logs,conf | pg+redis+kafka+cassandra | 注入全部依赖地址；先 install 后 run |
| wechat-messenger | wechat-messenger:v2.1.0 | 告警 | **手动** | — | logs,download | — | **不纳入自动编排**：现场手动配 webhook/阈值并启动（§14-14）|
| netdata | netdata:v2.5.3 | 监控 | 每节点 | 19999 | 命名卷 | — | 建议每台部署一份 |

### 16.3 放置矩阵（Placement Matrix）UI

- **矩阵视图**：行=节点，列=服务，单元格勾选/填实例。集群类服务（kafka/cassandra/redis-cluster）可在多个节点各放一个实例，自动编号 `kafka-01/02/03`。
- **校验规则**（放置时即时校验）：
  - 单例服务（pg/redis/iotcloud）全集群只能 1 个。
  - wechat-messenger：工具不自动编排，仅在矩阵中标记“手动”，由现场自行配置启动（§14-14）。
  - kafka/cassandra：要么 1 个单例，要么集群模式（节点数可变：3、5…，通常一节点一实例；成员数=放置实例数）。
  - redis-cluster：主节点 ≥3，从节点数为主节点整数倍（沿用 `redis-cluster.sh` 规则）。
  - iotcloud 需要 pg、redis、kafka、cassandra 都已在某处放置，否则报缺依赖。
  - netdata 建议每节点一份（可一键“全节点铺”）。
- **端口冲突检测**：同一节点上多个实例端口不得冲突（自动建议错开，如 kafka 多实例需独立端口或独占节点）。

### 16.4 每实例配置项

- **数据存储位置**：每个有状态实例可指定宿主机数据目录（联动 §7 步骤4 磁盘预览，推荐落大容量 SSD）。默认 `/opt/deploy-tool/services/<svc>/data`。这就是你要的“pg 指定数据存储位置”。
- **端口**：默认取目录值，冲突时可改。
- **密码**：pg(`POSTGRES_PASSWORD`)、redis(`requirepass`) **沿用安装包内默认值，不修改**（§14-15）。iotcloud `.env` 自动注入相同的默认密码，保证一致。
- **iotcloud 存储策略**：`DATABASE_TS_TYPE`(cassandra)、`DATABASE_TS_LATEST_TYPE`(redis/redis-cluster) 可选。

### 16.5 依赖装配引擎（配置生成）

为每个实例生成最终 compose/env（核心逻辑）：

- **iotcloud**：依赖地址**只通过生成 `.env` 注入**，不改 `thingsboard.yml`（详见 §17.6 与 §17.10 配置注入原则）。
- **cassandra 集群**：每实例设 `CASSANDRA_SEEDS=<seed节点IP列表>`、`CASSANDRA_LISTEN_ADDRESS=<本节点IP>`、`CASSANDRA_BROADCAST_ADDRESS=<本节点IP>`，`network_mode: host`；首启先起 seed 节点，待 `UN` 后再起其余，避免脑裂。
- **kafka KRaft 集群**：每实例设唯一 `KAFKA_CFG_NODE_ID`(0/1/2…)，`KAFKA_CFG_CONTROLLER_QUORUM_VOTERS=0@ip1:9093,1@ip2:9093,2@ip3:9093`，`KAFKA_CFG_ADVERTISED_LISTENERS=PLAINTEXT://<本节点IP>:9092`，共享同一 `KAFKA_KRAFT_CLUSTER_ID`，副本因子(`OFFSETS/TRANSACTION_*`)按 broker 数（≥3 设 3）。
- **redis-cluster**：按 `redis-cluster.sh` 思路生成各节点 compose（`ANNOUNCE_IP=<本节点IP>`、端口错开、host 网络），全部起来后执行 `--cluster create` 初始化主从。
- **单例 pg/redis**：基本沿用原 compose，仅替换数据卷路径/密码/端口。
- 装配产物：每个目标节点一份 `services/<svc>/docker-compose.yml` + `.env`，由本工具生成后下发。

### 16.6 镜像按需分发

- 每个节点**只接收该节点要跑的服务镜像**（如 node-02 只推 `kafka.tar` + `cassandra.tar`），按节点架构匹配。
- 分发前 `docker image inspect` 去重，已存在则跳过 load，节省 GB 级传输。
- 大镜像 SFTP 带进度、可中断续传；多节点并发。

### 16.7 部署执行顺序（编排）

工具按依赖拓扑分层执行，每层并发、层间等待健康：

```
1) 分发镜像 → docker load（各节点并发，仅各自所需镜像）
2) 起存储/中间件层：
   - postgres、redis / redis-cluster
   - cassandra：先 seed 后其余 → 等所有节点 UN
   - kafka：全 broker 起 → 等 KRaft 就绪
   - redis-cluster：全节点起 → 执行 cluster init（主从）
2.5) 【仅 cassandra 集群模式】预建 keyspace：执行
     `CREATE KEYSPACE IF NOT EXISTS thingsboard WITH replication={'class':'NetworkTopologyStrategy','datacenter1':3}`
     （在 iotcloud install 之前，确保 TB 不以 RF=1 建库；详见 §17.6）
3) 起应用层：
   - iotcloud：先 `docker-compose-install.yml`（一次性 install.sh 初始化 DB schema，阻塞等完成）
              再 `docker-compose.yml`（up -d 正式启动）
   - wechat-messenger、netdata
4) 健康校验：逐服务探活（端口/容器状态/集群状态），结果回 UI
5) 建 proxy 网络（步骤5 已建则跳过）
```

> iotcloud 的两段式启动来自现有 `start.sh`（`install-iotcloud` 先于 `iotcloud`），工具需保留此顺序，否则 DB 未初始化导致启动失败。

### 16.8 部署预览与确认

执行前生成**完整部署预览**：每节点将运行哪些容器、拉起哪些镜像、占用端口、数据卷落盘路径、依赖装配后的连接关系图。确认（warning 级；重新部署含删卷为 danger 级，见 §15）后才执行。

### 16.9 幂等、重部署与卸载

- 重复部署：先对目标服务 `down` 再 `up -d`（保留数据卷，除非用户显式选清数据）。
- 卸载：详见 **§20 一键卸载**（按服务/节点/集群粒度，danger 级确认，默认保留数据卷）。
- 部署清单持久化到工程（ClusterProject.stepState），下次打开可基于现状增量调整。

### 16.10 数据模型补充（步骤6）

```ts
interface ServicePlacement {
  service: string;          // postgres / kafka / cassandra ...
  instanceId: string;       // kafka-01
  nodeId: string;           // 放在哪台
  dataPath?: string;        // 宿主机数据目录（联动磁盘步骤）
  ports?: Record<string, number>;
  params?: Record<string, string>;  // 密码、副本因子、DATABASE_TS_TYPE 等
}

interface DeploymentPlan {
  placements: ServicePlacement[];
  // 由装配引擎推导：每实例最终 env、依赖连线、启动分层
  resolvedConfigs: Record<string /*instanceId*/, { compose: string; env: Record<string,string> }>;
  order: string[][];        // 分层启动顺序
}
```

## 17. 动态配置生成引擎（拓扑驱动）

> 解决“现有安装包只适配单机、多节点集群配置如何来”的问题。

### 17.1 为什么不直接改单机 compose（sed/正则）

直接对安装包里的单机 `docker-compose.yml` 做文本替换有三个坑：

- **脆弱**：YAML 缩进/引号/锚点稍变就替换错位，难维护。
- **多实例难表达**：3 个 kafka 各自的 `node-id`、`advertised` 不是“替换一处”能搞定，是“按拓扑批量生成”。
- **不可测**：文本 diff 难做单元测试。

**采用方式：把单机 compose 当“参考蓝本”，工具内置「参数化构建器」，按拓扑构建配置对象再序列化为 YAML。** 单实例时渲染结果≈原单机配置（仅替换路径/密码/端口）；集群时渲染集群变体。这样单机/集群同一套引擎、可单元测试、可预览 diff。

### 17.2 渲染流水线

```
放置矩阵(placements) + 服务目录(catalog) + 节点IP表
        │
        ▼  ① 计算集群拓扑（每服务）
   members[] / 每实例的 index / seeds / 共享ClusterId / 副本因子
        │
        ▼  ② 为每个实例调用对应 ServiceComposeBuilder(ctx)
   ctx = { instanceId, nodeIp, allMemberIps, index, dataPath, ports, passwords, sharedClusterId, mode:'single'|'cluster' }
        │
        ▼  ③ 构建 ComposeSpec 对象 → 用 yaml 库序列化
   每节点每服务一份 docker-compose.yml + .env
        │
        ▼  ④ 生成「配置预览」(可看到将下发的完整 YAML) → §15 确认 → 下发执行
```

- 构建器是**纯函数** `(ctx) => ComposeSpec`，输入拓扑上下文、输出配置对象，便于测试与预览。
- `mode='single'` 走单机分支（贴近原包）；`mode='cluster'` 走集群分支（注入拓扑参数 + host 网络）。
- 共享值（如 `KAFKA_KRAFT_CLUSTER_ID`）在拓扑计算阶段**生成一次**并存入 `DeploymentPlan`，所有成员复用；重部署时从工程读取保持稳定。

### 17.3 拓扑计算（关键中间产物）

```ts
interface ClusterTopology {
  service: 'kafka' | 'cassandra' | 'redis-cluster';
  members: { instanceId: string; nodeId: string; ip: string; index: number }[]; // index: 0,1,2...
  seeds: string[];          // cassandra: 取前 1-2 个成员 IP
  sharedClusterId?: string; // kafka KRaft cluster id（生成一次）
  replicationFactor: number;// = min(成员数, 3)
}
```

### 17.4 Kafka KRaft 集群模板（3 节点示例）

参考原单机配置，集群差异点全部由拓扑计算填充。第 `i` 个 broker（节点 IP=`IP_i`，共 N 个，IP 列表 `IP_0..IP_{N-1}`）：

```yaml
services:
  kafka:
    image: bitnami/kafka:3.9
    container_name: kafka                 # 单节点一实例，容器名固定即可
    network_mode: host                    # 集群模式用 host 网络（端口直通、advertised 真实 IP）
    environment:
      - KAFKA_ENABLE_KRAFT=yes
      - KAFKA_CFG_PROCESS_ROLES=controller,broker
      - KAFKA_CFG_NODE_ID={{i}}                                  # 每实例唯一: 0,1,2
      - KAFKA_KRAFT_CLUSTER_ID={{sharedClusterId}}               # 全集群同一个(生成一次)
      - KAFKA_CFG_CONTROLLER_QUORUM_VOTERS=0@IP_0:9093,1@IP_1:9093,2@IP_2:9093
      - KAFKA_CFG_LISTENERS=PLAINTEXT://:9092,CONTROLLER://:9093
      - KAFKA_CFG_ADVERTISED_LISTENERS=PLAINTEXT://IP_i:9092      # 本节点真实 IP
      - KAFKA_CFG_LISTENER_SECURITY_PROTOCOL_MAP=PLAINTEXT:PLAINTEXT,CONTROLLER:PLAINTEXT
      - KAFKA_CFG_CONTROLLER_LISTENER_NAMES=CONTROLLER
      - KAFKA_CFG_INTER_BROKER_LISTENER_NAME=PLAINTEXT
      # 副本因子按 broker 数提升(≥3 设 3)，保证高可用
      - KAFKA_CFG_OFFSETS_TOPIC_REPLICATION_FACTOR={{rf}}
      - KAFKA_CFG_TRANSACTION_STATE_LOG_REPLICATION_FACTOR={{rf}}
      - KAFKA_CFG_TRANSACTION_STATE_LOG_MIN_ISR={{rf-1}}
      - KAFKA_CFG_DEFAULT_REPLICATION_FACTOR={{rf}}
      - KAFKA_CFG_MIN_INSYNC_REPLICAS={{rf-1}}
    volumes:
      - {{dataPath}}:/bitnami/kafka
      - /etc/localtime:/etc/localtime:ro
    restart: always
```

- `sharedClusterId`：用 22 字符 base62（如原包 `abcdefghijklmnopqrstuv`）随机生成一次。
- 启动：N 个 broker 可并发起，KRaft 自动选举；工具等 controller quorum 就绪（探活 9092）再放行下一层。

### 17.5 Cassandra 集群模板（3 节点示例）

第 `i` 个节点（IP=`IP_i`），seeds 取前 1–2 个成员：

```yaml
services:
  cassandra:
    image: cassandra:4.1.3
    container_name: cassandra
    network_mode: host                    # gossip 7000 需对端用真实 IP 回连
    environment:
      - CASSANDRA_CLUSTER_NAME=cassandra
      - CASSANDRA_DC=datacenter1
      - CASSANDRA_RACK=rack1
      - CASSANDRA_ENDPOINT_SNITCH=GossipingPropertyFileSnitch
      - CASSANDRA_NUM_TOKENS=256
      - CASSANDRA_SEEDS=IP_0,IP_1                       # 1-2 个种子节点
      - CASSANDRA_LISTEN_ADDRESS=IP_i                   # 本节点真实 IP
      - CASSANDRA_BROADCAST_ADDRESS=IP_i
      - CASSANDRA_BROADCAST_RPC_ADDRESS=IP_i
      - CASSANDRA_RPC_ADDRESS=0.0.0.0
    volumes:
      - {{dataPath}}:/var/lib/cassandra
      - {{logPath}}:/var/log/cassandra
      - /etc/localtime:/etc/localtime:ro
    restart: unless-stopped
```

- **启动顺序很关键（不可并发）**：Cassandra 集群必须**串行 bootstrap**——先起全部 seed 节点，`nodetool status` 等到 `UN`，再**逐个**起非 seed 节点（每个等到 `UN` 再起下一个），避免 token 冲突/脑裂。工具的编排引擎对 cassandra 走“串行+健康门禁”特殊分层。
- **副本因子**：TB 在 cassandra 建 keyspace 的副本因子需配成 ≥3（对应 §17.6 的 iotcloud 参数），否则集群虽起但数据不冗余。

### 17.6 iotcloud 依赖地址注入（只写 .env，绝不改 thingsboard.yml）

`thingsboard.yml` 内全部为 `${VAR:default}` 占位，运行时由 `.env`（compose `env_file: ./.env`）覆盖。因此工具**只生成/追加 `.env`，conf 目录与 thingsboard.yml 原样下发不改动**。变量名均已对照 `conf/thingsboard.yml` 核实。

iotcloud 单机连“集群版”依赖时，生成的 `.env`（在原有 `APP_IMAGE=...` 基础上追加）：

```ini
# === 镜像（原包自带，保留） ===
APP_IMAGE=iotcloud:4.1.0-20260530.1

# === postgres（单例 → 其所在节点 IP）===
SPRING_DATASOURCE_URL=jdbc:postgresql://<pgIP>:5432/thingsboard
SPRING_EVENTS_DATASOURCE_URL=jdbc:postgresql://<pgIP>:5432/thingsboard_events
SPRING_DATASOURCE_PASSWORD=JvUcMbDxjYY4M8sj    # 沿用包内 pg 默认密码（§14-15，不修改）

# === redis（单例 → 其所在节点 IP）===
REDIS_HOST=<redisIP>
REDIS_PORT=6379
REDIS_PASSWORD=eRLvW23KYiAakR                  # 沿用包内 redis 默认密码（§14-15，不修改）

# === cassandra 集群（CASSANDRA_URL 支持逗号分隔多 contact point）===
CASSANDRA_URL=<casIP0>:9042,<casIP1>:9042,<casIP2>:9042
CASSANDRA_CLUSTER_NAME=cassandra
CASSANDRA_LOCAL_DATACENTER=datacenter1
DATABASE_TS_TYPE=cassandra

# === kafka 集群（KRaft，无需 zookeeper）===
TB_QUEUE_TYPE=kafka
ZOOKEEPER_ENABLED=false
TB_KAFKA_SERVERS=<kfkIP0>:9092,<kfkIP1>:9092,<kfkIP2>:9092
TB_QUEUE_KAFKA_REPLICATION_FACTOR=3            # = broker 数（≥3 设 3），保证队列高可用
```

> 变量名已核实于 `thingsboard.yml`：`SPRING_DATASOURCE_URL`/`SPRING_EVENTS_DATASOURCE_URL`、`REDIS_HOST`/`REDIS_PORT`、`CASSANDRA_URL`/`CASSANDRA_CLUSTER_NAME`/`CASSANDRA_LOCAL_DATACENTER`、`TB_KAFKA_SERVERS`/`TB_QUEUE_TYPE`/`TB_QUEUE_KAFKA_REPLICATION_FACTOR`/`ZOOKEEPER_ENABLED`。
>
> ⚠️ **cassandra keyspace 副本因子（仅集群模式需要）**：TB 默认以 RF=1 建 `thingsboard` keyspace，3 节点集群下每行只存 1 份，挂 1 台即丢约 1/3 数据，等于没有冗余。要真正高可用须让该 keyspace RF=3。
>
> **采用「预建 keyspace」方案（推荐，无需 ALTER/repair）**：在 cassandra 集群就绪后、**iotcloud `install` 之前**，工具先执行一次：
> ```sql
> CREATE KEYSPACE IF NOT EXISTS thingsboard
>   WITH replication = {'class':'NetworkTopologyStrategy','datacenter1':3};
> ```
> TB 安装脚本用的是 `CREATE KEYSPACE IF NOT EXISTS`，已存在则不覆盖，于是直接采用我们预设的 RF=3。**因建库时尚无数据，无需 `ALTER`，也无需 `nodetool repair`**，全程不触碰 thingsboard.yml。
>
> 兜底：若校验发现 keyspace 已被以 RF=1 建出（如顺序异常），再走 `ALTER KEYSPACE ... datacenter1:3` + `nodetool repair` 补救。
> 单节点 cassandra：RF=1 正常，**工具不做任何 keyspace 操作**。
> （待核对：确认该 TB 版本建 keyspace 确为 `IF NOT EXISTS`，见 §14-11。）

### 17.7 防火墙/端口连通（现场负责，工具不操作）

**工具不修改任何防火墙规则**。host 网络下集群端口需节点间互通，由**现场预先统一关闭防火墙（或自行放通）**。工具的职责仅限：
- 部署前**探测端口可达性**（如从 A 节点探 B 节点的 9092/9093/7000/9042），不可达则在 UI 明确提示“请现场关闭防火墙/放通端口”，**不自动改动**。

涉及端口参考：kafka `9092/9093`、cassandra `7000/9042`、redis-cluster `700x + bus`、各服务对外端口见 §16.2。

### 17.8 单机 vs 集群的统一抽象

| 服务 | 单机模式(mode=single) | 集群模式(mode=cluster) |
|---|---|---|
| kafka | 原单机配置（bridge+proxy，`kafka:9092`） | 本节 §17.4，host 网络、N broker |
| cassandra | 原单机配置（bridge+proxy） | 本节 §17.5，host 网络、串行 bootstrap |
| redis-cluster | —（用单机 redis 即可） | 沿用 `redis-cluster.sh` 思路渲染 + init |
| pg/redis/iotcloud/wechat/netdata | 始终单机渲染（替换路径/密码/端口/依赖地址） | 同左 |

放置矩阵里某服务被放到 **≥2 个节点** 即判定为 cluster 模式，自动切换到集群构建器；放 1 个则单机模式。

### 17.9 配置预览（重要）

下发前，UI 展示每个节点每个服务**最终渲染出的完整 YAML + env**（可折叠、可与原单机模板做 diff），让你在执行前就能逐字核对集群参数对不对，再走 §15 确认。这是“动态生成”能被信任的关键。

### 17.10 配置注入原则（统一约束）

适用于所有服务，尤其 iotcloud：

1. **不改应用配置文件**：`thingsboard.yml`、`logback.xml` 等 conf 文件**原样从安装包下发**，工具一律不修改。所有差异化只通过 **`.env`（或 compose 的 `environment`）覆盖** `${VAR:default}` 占位实现。
2. **生成物仅三类**：① 各服务的 `docker-compose.yml`（由 §17 构建器渲染）② 各服务的 `.env`（依赖地址/密码/集群参数）③ 必要的一次性运维命令（如 cassandra `ALTER KEYSPACE`、redis `cluster create`）。
3. **conf 目录透传校验**：下发后校验 conf 内容与包内一致（校验和），确保未被误改。
4. **密码一致性**：pg/redis 实例的密码与 iotcloud `.env` 中对应变量由同一数据源生成，避免“改了实例密码但 iotcloud 没同步”。
5. **可回溯**：每次生成的 `.env`/compose 存入工程快照，重部署可比对 diff。

## 19. 发行版适配层（OsAdapter，本期实现 Ubuntu/Debian）

现场以 Ubuntu 为主，但**代码保留发行版抽象**，未来加麒麟/统信/CentOS 只需新增一个适配器、不改业务逻辑。

```ts
interface OsAdapter {
  id: 'debian' | 'rhel' | 'kylin' | 'uos' | ...;
  match(probe: NodeProbe): boolean;            // 由 /etc/os-release ID/ID_LIKE 判定
  installLocalPackage(files: string[]): string; // deb: `dpkg -i`; rhel: `rpm -ivh`
  chronyPackage(): string;                      // 返回该发行版/架构的 chrony 离线包名
  serviceManager: 'systemd';                    // 统一 systemctl（本期目标均 systemd）
  setHostname / setTimezone / ...               // 命令差异收敛在适配器内
}
```

- **本期只实现 `DebianAdapter`**（apt/dpkg、deb 包）；步骤1 探测到非 Debian 系时提示“暂不支持，待扩展”。
- 业务步骤（2/3/5/6）只调用 `OsAdapter` 的抽象方法，**不写死发行版命令**，保证扩展不动主干。
- 新发行版接入清单：①写 `match` ②给 chrony 离线包 ③（如有）覆盖差异命令。

## 20. 一键卸载（§14-16）

复用安装包 `uninstall.sh` 思路，做成图形化、可选范围、带 danger 确认的卸载能力。

### 20.1 卸载粒度

| 粒度 | 动作 | 级别 |
|---|---|---|
| 单服务 | 对目标节点上某服务 `docker-compose down`（可选 `-v` 删卷） | warning / danger(删卷) |
| 单节点 | 该节点上本工具部署的全部服务 down + 清理 compose/.env | danger |
| 整集群 | 所有节点全部服务 down；可选删数据卷 | danger |
| 卸载 docker | 停止 docker、移除二进制/`docker.service`/`/etc/docker`、可选删 `data-root` | danger（最高）|

### 20.2 规则与安全

- **默认保留数据卷**：down 不带 `-v`；删数据需用户**显式勾选 + 输入确认词**（§15 danger）。
- **依赖顺序倒序**：先停应用层（iotcloud/wechat），再停中间件/存储（kafka/cassandra/redis/pg），避免应用报错刷屏。
- **只动“本工具部署的”**：通过工程快照（§17.10-5）记录的容器/路径精确定位，不误删现场其它容器。
- **卸载预览**：执行前列出将停止的容器、将删除的卷/目录、受影响节点，确认后执行。
- **幂等**：可重复卸载；目标已不存在则跳过并标记“已是干净状态”。

### 20.3 与“强制重装”的关系

步骤5 的“强制重装 docker”（§7 步骤5）本质是“卸载 docker + 重装”，复用本节卸载逻辑 + 步骤5 安装逻辑串联。
