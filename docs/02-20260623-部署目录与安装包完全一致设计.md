# 02 · 部署目录与安装包完全一致 — 设计

> 顺序编号：02

> 关联：主设计 `docs/设计文档.md`（§16/§17）、`docs/01-…`

---

## 1. 目标

部署后，每个节点的 `~/sprixin-iotcloud/` 目录**与安装包内 `sprixin-iotcloud/` 结构完全一致**，包含：

```
~/sprixin-iotcloud/
├─ docker/                 # 离线 docker 物料（bin/、service/、daemon.json）
├─ images/                 # 镜像 tar
├─ services/               # 各服务目录（原始 compose + 工具生成/覆盖）
│  ├─ postgres/ redis/ kafka/ cassandra/ redis-cluster/
│  ├─ netdata/ wechat-messenger-v2.1.0/ iotcloud/(conf、.env…)
├─ install.sh  start.sh  uninstall.sh  .svninfo.txt
```

在此基础上，工具对**需要集群化/注入的服务**做最小覆盖（见 §3），其余文件与包**逐字节一致**。

---

## 2. 方案：整包落地 + 最小覆盖

**两阶段：**

1. **整包落地**：把安装包的 `sprixin-iotcloud/` 整树放到节点 `~/`，得到与包一致的结构。
2. **最小覆盖**：仅对下列文件按 §17 重写（集群/注入必需），其余不动：
   - 集群模式的 `services/kafka/docker-compose.yml`、`services/cassandra/docker-compose.yml`（host 网络 + 真实 IP）。
   - `services/iotcloud/.env`（注入 pg/redis/kafka/cassandra 真实地址，不改 thingsboard.yml）。
   - 单机模式：**直接用包内原始 compose**（无需覆盖），数据目录命名天然与包一致（kafka `kafka_0_data`、cassandra `cassandra_node1_data` 等）。

> 好处：单机服务"零改动、纯原包"；只有真正要集群的服务才被工具接管，最大化"一致性"。

---

## 3. 覆盖清单（哪些文件会被工具改写）

| 文件 | 何时改写 | 原因 |
|---|---|---|
| `services/kafka/docker-compose.yml` | kafka 多实例(集群) | KRaft node-id/quorum/advertised + host 网络 |
| `services/cassandra/docker-compose.yml` | cassandra 多实例(集群) | seeds/listen + host 网络 |
| `services/iotcloud/.env` | 总是 | 注入依赖真实地址（§17.6） |
| 其余（pg/redis/单机 kafka/单机 cassandra/netdata/脚本/docker/） | 不改 | 与包逐字节一致 |

数据目录在 `up` 前 `mkdir -p` + `chmod 777`（kafka/cassandra 容器非 root）。

---

## 4. 待确认：images 的取舍（影响传输量，需你拍板）

整包里 `images/` 有 7 个镜像 tar（合计约 1.5G）。两种做法：

- **A. 完全复刻（含全部镜像）**：每个节点 `~/sprixin-iotcloud/images/` 都放**全部 7 个 tar**。真正"完全一致"，但**每节点都要传 ~1.5G**（3 节点≈4.5G、5 节点≈7.5G），且每节点会 `docker load` 用不到的镜像。
- **B. 结构完全一致，镜像按需（推荐）**：`docker/`、脚本、`services/` 全部一致；但 `images/` 只放**该节点实际要跑的服务镜像**（node-02 只放 kafka/cassandra）。省大量流量，结构仍一致、只是 images 目录内文件按需。

> 其余目录（docker/、services/、脚本）两种方案都完全一致；差别只在 images/ 是否每台都全量。

---

## 5. 实现要点

- **方案 A**：把整个安装包 `tar.gz` 上传到节点，远端 `tar xzf` 解到 `~/`（得到逐字节一致的整树），再覆盖 §3 的少数文件。最省事、最一致。
- **方案 B**：本地把包解成整树（images 留空或按需），`putDir` 上传 `docker/`+`services/`+脚本到 `~/sprixin-iotcloud/`，再单独把该节点所需镜像 tar 放进 `~/sprixin-iotcloud/images/`，最后覆盖 §3 文件。
- 覆盖：渲染引擎对集群服务/iotcloud 生成内容写回对应路径。
- **幂等**：可重复部署；整包落地用覆盖式（已存在则更新），不破坏已运行容器的数据目录。
- **卸载（§20/01）**：路径不变（`~/sprixin-iotcloud/services/<svc>`），现有卸载逻辑兼容。

---

## 6. 边界与风险

- 方案 A 传输量大（每节点 ~1.5G+docker），离线现场网络要扛得住；可加并发与续传。
- `docker/` 物料：步骤5 已上传到 `/opt/deploy-tool/docker` 用于装 docker；本设计在 `~/sprixin-iotcloud/docker` 再放一份是为"与包一致"（冗余但无害）。可选：步骤5 改为也从 `~/sprixin-iotcloud/docker` 安装，避免两份。
- 覆盖文件需在"整包落地"之后执行，避免被原始文件盖回。
