## 1. 目标与定位

从"一次性单集群部署工具" → "**多集群管理平台**"：

- 打开应用先进入**集群列表**，按业务**分组**管理多个集群（如「日新鸿晟」「一汽现场」）。
- 每个集群下挂各自的物理机节点（A 组 3 台、B 组 1~10 台不等）。
- 点进某个集群后，既能走**部署向导**（新装/重部署），也能进**运维总览**（管理已有集群）。
- 既支持"从零部署"，也支持"纳管现有集群"（加节点 → 直接看运维总览，靠真实 docker ps 发现）。

> 核心思路：现有单集群能力（向导 6 步 + 运维总览）几乎原样复用，只在外层加一层"集群"作用域。一个**集群 = 一组节点 + 该集群的部署/编排状态**。

---

## 2. 信息架构 / 导航层级

```
应用
├─ [集群列表]  ← 启动落地页
│   ├─ 集群卡片：日新鸿晟（3 节点）
│   ├─ 集群卡片：一汽现场（10 节点）
│   └─ [+ 新建集群]
│
└─ 进入某集群 → [集群工作区]（顶栏显示集群名 + 返回集群列表）
    ├─ 部署向导（6 步，作用于该集群节点）
    └─ 运维总览（该集群节点的状态/容器/一键全卸载）
```

- 顶层视图 `appView`：`clusters`（列表）| `cluster`（进入某集群）。
- 进入集群后，沿用现有的「部署向导 / 运维总览」切换（`view`），但作用域是当前集群。
- 顶栏新增「← 集群列表」返回入口与当前集群名。

## 3. 数据模型

一个集群即"带 id/名称的工程"（现有 `ClusterProject` 已含 nodes/packages/stepState）：

```ts
interface Cluster {
  id: string                 // 内部唯一 id
  name: string               // 分组名，如「日新鸿晟」
  remark?: string
  nodes: NodeConfig[]        // 该集群的节点
  packages: InstallerPackage[]
  stepState: Record<string, unknown>  // 含 placements 等
  createdAt: string
  updatedAt: string
}

interface ClusterSummary {   // 列表页用，轻量
  id: string
  name: string
  remark?: string
  nodeCount: number
  updatedAt: string
}
```

- 密码仍用 `safeStorage` 加密落盘（明文不落盘）；跨机迁移用步骤1 的「导入导出」（docs/05）。

## 4. 持久化

- 现状：单文件 `userData/projects/default.json`。
- 改为：**每集群一个文件** `userData/clusters/<clusterId>.json`（结构=Cluster，密码加密）。列表页扫描该目录生成 `ClusterSummary[]`。
- **平滑迁移**：启动时若发现旧的 `projects/default.json` 且 `clusters/` 为空，自动转换为一个名为「默认集群」的集群，不丢历史配置。

| 模块 | 改动 |
|---|---|
| `store/ConfigStore.ts` | `loadProject/saveProject` → `listClusters / loadCluster(id) / saveCluster / createCluster / renameCluster / deleteCluster`；旧文件迁移 |

| 运维总览/各步骤 | 不变（仍读 store.nodes，等于"当前集群"的节点）|

## 6. 集群列表页 UI（ClusterList）

- 卡片网格，每张：集群名、节点数、备注、更新时间；操作：进入 / 重命名 / 删除。
- 顶部「+ 新建集群」：填名称（可选备注）→ 创建空集群 → 进入其步骤1 添加节点。
- 进入集群（§14-2）：**已部署过 → 直接进运维总览；未部署过 → 进部署向导步骤1**。判定"已部署过"以集群的 `stepState.placements` 非空为准（无需在进入前 SSH 探测）。

> 列表页保持轻量：只展示元数据，不在列表页对所有集群批量探测（避免一打开就 SSH 风暴）；状态在进入集群后再探测。

---

## 7. 集群工作区导航

- 顶栏：`离线集群部署工具 / [集群名] | 部署向导 · 运维总览 ……  [← 集群列表]`。
- 「← 集群列表」：保存当前集群 → `appView=clusters`。
- 切换集群前自动保存，避免丢配置。
- `busy`（执行中）期间禁用"返回集群列表"和集群切换（与现有导航锁一致）。

## 9. 边界与风险

- **删除集群（§14-3）**：danger 级二次确认；**仅删本地集群配置文件，绝不触碰远端机器/服务**（弹窗明确说明）。
- **跨集群同 IP（§14-4）**：**允许**同一 IP 出现在多个集群；新建/添加节点时若该 IP 已存在于其它集群，**给出提醒**（"该 IP 也在『一汽现场』中"），但不阻断。运维总览各看各的容器互不影响。
- **当前集群未保存**：切换/返回前自动保存。
- **旧数据迁移**：只在首次、且 clusters/ 为空时迁移，幂等。
- **节点规模**：B 组可能 10 台，列表/总览需支持较多节点（现有折叠/并发已具备）。

---

## 10. 实施计划

1. **M8-A 持久化层**：ConfigStore 多集群 + 迁移；`clusters:*` IPC。
2. **M8-B 状态层**：wizard store 加 `appView/clusterId` + open/close 集群。
3. **M8-C 列表页**：`ClusterList.tsx` 增删改查 + 进入；App 顶层路由 + 顶栏返回。
4. **M8-D 串联验证**：建两个集群、各加节点、分别进向导/总览，互不串数据。

---

## 11. 决策（已确认 2026-06-23）

1. ~~层级~~ → **集群=分组（单层）**：集群即一组节点，无父分组。
2. ~~进入集群默认落地~~ → **已部署过(placements 非空)→运维总览；未部署→部署向导步骤1**。
3. ~~集群删除~~ → **仅删本地集群配置，不影响远程物理节点/服务**（danger 二次确认）。
4. ~~节点跨集群~~ → **允许同一 IP 跨集群**，但添加时若已存在于其它集群则**提醒**（不阻断）。
